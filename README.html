一、实现原理

1.1 什么是 Page Cache

Linux 内核在读写磁盘文件时，会将文件数据缓存在内存中，这部分内存称为 Page Cache（页缓存）。当进程再次访问同一文件时，内核直接从内存返回数据，避免磁盘 I/O，大幅提升性能。

Page Cache 是透明的——用户态程序无法直接操作它，但可以通过 mincore(2) 系统调用查询哪些页在缓存中。pgcache 的核心就是利用这一机制。

Page Cache 的生命周期

  • 读文件时：内核将磁盘数据拷贝到 page cache，然后返回给用户态
  • 再次读同一文件：直接从 page cache 返回，不触发磁盘 I/O
  • 内存不足时：内核回收最久未使用的缓存页（类似 LRU）
  • 文件被修改：脏页（dirty page）会定期刷回磁盘

1.2 mincore 系统调用

int mincore(void *addr, size_t length, unsigned char *vec);

mincore 是整个工具的核心。它查询一段虚拟地址区间内哪些页驻留在物理内存（page cache）中。工作流程如下：

open(file)
打开目标文件
→
fstat(fd)
获取文件大小
→
mmap()
映射到虚拟地址空间
→
mincore()
查询每页缓存状态
→
统计 + 输出
计算缓存百分比

详细步骤

步骤系统调用说明
1. 打开文件 open(path, O_RDONLY) 以只读方式打开目标文件，获得文件描述符 fd
2. 获取大小 fstat(fd, &st) 通过 st_size 获取文件字节数，跳过目录和零大小文件
3. 内存映射 mmap(NULL, size, PROT_NONE, MAP_SHARED, fd, 0) 将文件映射到虚拟地址空间
PROT_NONE：不设置读写权限，仅用于查询
MAP_SHARED：与 page cache 共享同一份物理页
4. 查询缓存 mincore(addr, size, vec) 内核为每一页（通常 4KB）写入一个字节到 vec
LSB = 1 表示该页在 page cache 中
LSB = 0 表示该页不在
5. 统计 遍历 vec[i] & 1 统计在缓存中的页数，计算 percent = cached/pages × 100
6. 清理 munmap() + close() 解除映射，关闭文件描述符

mincore 向量示意

文件大小 = 16KB，页大小 = 4KB → 4 页

vec[0] = 0x01  →  第 1 页在缓存中 ✓
vec[1] = 0x00  →  第 2 页不在缓存 ✗
vec[2] = 0x01  →  第 3 页在缓存中 ✓
vec[3] = 0x01  →  第 4 页在缓存中 ✓

cached = 3,  pages = 4,  percent = 75.0%
为什么用 mmap + MAP_SHARED？
MAP_SHARED 使得虚拟地址空间中的映射与内核的 page cache 共享同一份物理内存页。 因此 mincore 查询的就是 page cache 的真实状态，而不是一份独立的副本。

1.3 多进程文件收集（--top 模式）

--top N 需要回答一个问题：系统上所有被映射的文件有哪些？

答案来自每个进程的 /proc/[pid]/maps 文件。

/proc/[pid]/maps 文件格式

地址范围                  权限  偏移      设备    inode  路径名
00400000-00452000        r--p 00000000  08:01   12345  /usr/bin/vim
7f1234567000-7f12345a8000 r--p 00000000  08:01   67890  /usr/lib/libc.so.6

第 6 列（以 / 开头）就是被映射的文件路径。pgcache 的收集流程：

枚举所有 PID
读 /proc/ 数字目录
→
过滤 RSS=0
跳过内核线程
→
读 maps
提取文件路径
→
去重
qsort + 相邻去重
→
mincore
检测每个文件
→
排序 + Top N
按缓存降序

为什么要做去重？

同一个共享库（如 libc.so.6）会被几十个进程同时映射。如果不对路径去重，mincore 会重复检测同一个文件，浪费时间和 CPU。

去重策略：排序 + 相邻去重。先用 qsort 对所有路径按字典序排序，再线性扫描去掉相邻重复项。时间复杂度 O(n log n)，无需哈希表。

1.4 Mount Namespace 切换

在容器（Docker/LXC）环境中，进程看到的文件路径是相对于其 mount namespace 的。例如容器内的 /app/data.db 在宿主机上可能是 /var/lib/docker/overlay2/.../diff/app/data.db。

如果不切换 namespace 直接读 /proc/[pid]/maps，看到的路径在宿主机上可能不存在。pgcache 通过 setns() 解决这个问题：

int fd = open("/proc/[pid]/ns/mnt", O_RDONLY);
setns(fd, 0);   // 切换到目标进程的 mount namespace
close(fd);
注意：setns() 需要 CAP_SYS_ADMIN 权限。普通用户执行时该操作会失败，但 pgcache 采用 best-effort 策略——失败时静默跳过，继续处理下一个进程。

1.6 关键数据结构

cache_entry_t — 文件缓存状态

typedef struct {
    char    *name;       // 文件路径 (strdup 分配, 调用者负责 free)
    int64_t  size;       // 文件字节数
    time_t   timestamp;  // mincore 检测时间 (Unix 时间戳)
    time_t   mtime;      // 文件最后修改时间
    int      pages;      // 总页数 = ceil(size / pagesize)
    int      cached;     // 在 page cache 中的页数
    int      uncached;   // 不在缓存中的页数
    double   percent;    // cached / pages × 100.0
    char    *ppstat;     // 逐页状态: 每个字节 0/1 (仅 --histo/--pps 时分配)
} cache_entry_t;

entry_list_t — 动态数组（倍增扩容）

typedef struct {
    cache_entry_t *entries;
    int            count;      // 当前元素数
    int            capacity;   // 已分配容量
} entry_list_t;

// 扩容逻辑: capacity = 0 → 64, 否则 capacity *= 2
void list_push(entry_list_t *list, const cache_entry_t *entry) {
    if (list->count == list->capacity) {
        int new_cap = list->capacity == 0 ? 64 : list->capacity * 2;
        list->entries = realloc(list->entries, new_cap * sizeof(cache_entry_t));
        list->capacity = new_cap;
    }
    list->entries[list->count++] = *entry;
}

opts_t — 命令行选项

typedef struct {
    int pid;              // --pid PID (0 = 未设置)
    int top;              // --top N   (0 = 未设置)
    int nohdr;            // --nohdr   隐藏表头
    int bname;            // --bname   只显示 basename
    int terse;            // --terse   CSV 格式
    int json;             // --json    JSON 格式
    int histo;            // --histo   直方图
    int unicode;          // --unicode Unicode 边框表格
    int pps;              // --pps     JSON 含逐页状态
    int plain;            // --plain   纯文本无边框
} opts_t;

2.6 注意事项

需要 root 权限 才能读取其他进程的 /proc/[pid]/maps 和切换 mount namespace。普通用户只能查看自己进程的信息。

详细注意事项

  1. --top 扫描较慢 — 需要对系统上所有映射文件执行 mincore。文件数量取决于进程数和每个进程加载的共享库数量。一台典型的 Linux 桌面系统可能有数千个唯一文件。
  2. 结果是瞬时快照 — 进程可能随时创建/退出，page cache 随时变化。结果反映执行 mincore 时刻的状态，不代表过去或未来。
  3. 大文件 mmap 可能失败 — 32 位系统上虚拟地址空间有限（通常 3GB 用户空间），超大文件（如几个 GB 的数据库文件）可能无法映射，会被跳过。
  4. Deleted files 会被跳过 — /proc/[pid]/maps 中标记为 (deleted) 的文件无法打开（文件已被删除但进程仍持有句柄），pgcache 会自动跳过。
  5. Cached Size 是估算值 — 通过 Size × Percent / 100 计算。实际缓存的字节数取决于页对齐（最后一页可能只缓存了部分字节）。
  6. 百分比精度 — 文件的最后一页可能不满 4KB，但 mincore 仍然报告整页状态。因此对小文件或非页对齐大小的文件，百分比可能略有偏差。
